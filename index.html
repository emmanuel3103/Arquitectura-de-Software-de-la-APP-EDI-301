1. Introducción

1.1 Propósito

El presente documento describe la arquitectura del sistema EDI-301, diseñado para optimizar la gestión del programa de familias EDI de la Universidad Linda Vista. Su propósito es proporcionar una visión estructurada de los componentes, la interacción entre ellos y las decisiones técnicas adoptadas.

1.2 Alcance

La arquitectura abarca la aplicación móvil desarrollada en Flutter, la API REST construida en Node.js/Express y la base de datos SQL Server. Incluye aspectos de seguridad, despliegue y comunicación entre capas. Quedan fuera los procesos administrativos internos no informatizados del programa.

2. Representación arquitectónica

El sistema adopta un modelo cliente-servidor de tres capas, estructurado de la siguiente forma:

  1. Presentación: interfaz móvil en Flutter, con comunicación HTTPS.
  2. Lógica de negocio: API REST Node.js/Express, controladores y servicios.
  3. Datos: base de datos SQL Server con acceso mediante consultas parametrizadas.

La arquitectura sigue el modelo 4+1 de Kruchten, considerando vistas lógica, de proceso, de despliegue, de desarrollo y de casos de uso.

3. Objetivos y restricciones arquitectónicas

3.1 Objetivos

  • Garantizar modularidad y mantenibilidad.
  • Facilitar escalabilidad y rendimiento.
  • Aplicar autenticación y seguridad mediante JWT y TLS.
  • Asegurar compatibilidad multiplataforma (Android, iOS).

3.2 Restricciones

  • Uso obligatorio de Node.js, Flutter y SQL Server.
  • Cumplimiento de estándares institucionales de documentación.
  • Tiempo máximo de respuesta: 2 segundos por solicitud REST.

5. Vista lógica

5.1 Estructura y capas

  • Presentación (Flutter): pantallas, controladores y formularios conectados por HTTPS.
  • Negocio (Node.js/Express): controladores, servicios, repositorios y middleware.
  • Datos (SQL Server): tablas relacionales normalizadas, vistas y procedimientos almacenados.

Cada capa se comunica mediante objetos DTO y peticiones REST estandarizadas.

7. Vista de despliegue

7.1 Dispositivos móviles Android

Los usuarios acceden al sistema EDI-301 desde teléfonos o tabletas con sistema operativo Android. La aplicación se instala desde la tienda oficial o mediante distribución interna y se comunica con la API REST a través de HTTPS. El procesamiento pesado se realiza en el servidor, por lo que el dispositivo actúa principalmente como cliente de presentación.

7.2 Dispositivos móviles iOS

De forma equivalente, la aplicación puede ejecutarse en dispositivos iOS (iPhone/iPad). La comunicación con el backend se realiza también vía HTTPS, utilizando los mismos endpoints y formato de intercambio de datos (JSON), garantizando así la compatibilidad y coherencia funcional entre plataformas.

7.3 Servidor de API (Backend)

El servidor de API ejecuta la aplicación Node.js/Express, expone los servicios REST consumidos por los clientes móviles y aplica las reglas de negocio, autenticación (JWT) y validación de datos. Este servidor se encuentra detrás de un proxy inverso (por ejemplo, Nginx) que gestiona TLS, el enrutamiento y el registro de accesos.

7.4 Servidor de base de datos

La base de datos SQL Server se despliega en un servidor dedicado o instancia gestionada dentro de la infraestructura institucional. El backend accede a la base de datos mediante un pool de conexiones y consultas parametrizadas, manteniendo la integridad de la información de usuarios, familias, eventos y publicaciones.

7.5 Servicio de notificaciones push

Para el envío de notificaciones (por ejemplo, recordatorios de eventos o nuevas publicaciones), el sistema se integra con un servicio de notificaciones push (como Firebase Cloud Messaging). El backend registra los tokens de los dispositivos y envía mensajes cuando ocurren eventos relevantes, permitiendo mantener informados a los usuarios incluso cuando la aplicación no está en primer plano.

8. Tamaño y rendimiento

  • Usuarios concurrentes: hasta 1000.
  • Tiempo promedio de respuesta: 1.5 segundos.
  • Pooling SQL activo con 10 conexiones simultáneas.
  • Compresión HTTP y caché configurado en Nginx.
  • Base de datos estimada: 1 GB anual.

Conclusión

El diseño de arquitectura del sistema EDI-301 establece la base técnica y estructural para su desarrollo, garantizando un equilibrio entre rendimiento, seguridad y mantenibilidad. Su estructura en capas, uso de servicios REST y separación lógica por responsabilidades aseguran un crecimiento sostenible y una integración futura con nuevos módulos o servicios institucionales.
